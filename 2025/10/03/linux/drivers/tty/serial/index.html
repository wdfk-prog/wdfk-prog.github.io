drivers/serial 串行驱动核心(Serial Driver Core) 内核与UART硬件通信的统一框架

历史与背景

这项技术是为了解决什么特定问题而诞生的？

Serial（串行）驱动框架的诞生是为了解决一个在计算机历史上长期存在的核心问题：如何为各种不同的UART（通用异步收发器）硬件提供一个统一、标准、可移植的软件接口。

UART是实现串行通信（如RS-232标准）的基础硬件，但不同厂商、不同年代的UART芯片，其寄存器布局、功能特性（如FIFO深度）和编程方式都存在差异。如果没有一个统一的框架：

  1. 驱动代码无法复用：内核开发者需要为每一种新的UART芯片编写一个完全独立的、从头到尾的驱动程序。
  2. 上层接口不统一：每个驱动可能会向上层（TTY子系统）暴露不同的行为，或者需要上层进行特殊的处理。
  3. 设备驱动与硬件强耦合：应用程序或内核的其他部分会与特定的硬件实现绑定，降低了整个系统的可移植性。

Serial驱动框架的核心目标就是创建一个抽象层，将这些硬件差异隐藏起来。它定义了一套标准的接口和数据结构，使得具体的UART硬件驱动（如经典的8250/16550A系列芯片驱动）可以作为“插件”注册到这个框架中，而框架本身则负责与更上层的TTY子系统进行标准化的交互。

它的发展经历了哪些重要的里程碑或版本迭代？

  • 早期的独立驱动：在框架出现之前，内核中存在一些针对特定串口硬件的、独立的TTY驱动。
  • 8250驱动的通用化：PC体系结构中广泛使用的8250/16550A系列UART芯片的驱动（drivers/tty/serial/8250/）是这个框架事实上的先行者和范本。它本身就发展成了一个小型的框架，能够通过很少的平台特定代码来支持大量兼容或类似的UART硬件。
  • Serial Core的正式确立：内核最终将这种模式正式化，形成了drivers/serial/serial_core.c。这个核心文件定义了struct uart_driver和struct uart_port等关键数据结构，彻底将驱动逻辑（uart_driver）和端口实例（uart_port）分离开来。这允许一个通用的驱动（如8250驱动）可以管理任意数量的、具有不同硬件地址和中断的物理端口。
  • 与非传统串口的集成：框架的设计使其也能够支持非传统意义上的串口，例如USB-to-Serial适配器。这些设备的驱动（drivers/usb/serial/）最终也会注册一个uart_port，从而无缝地接入到TTY子系统中，被用户当作标准的/dev/ttyUSBx设备来使用。

目前该技术的社区活跃度和主流应用情况如何？

Serial驱动框架是Linux内核中一个极其基础和稳定的部分。它不是一个经常发生剧烈功能变化的领域，但由于几乎所有的嵌入式SoC都包含UART控制器，因此社区一直在为新的硬件平台积极地添加和维护相应的驱动。

  • 主流应用：
    • 内核调试控制台（Kernel Console）：这是其最重要的用途之一。在系统启动早期或崩溃时，内核通过串口输出调试信息，是开发者进行底层调试的生命线。
    • 嵌入式系统和物联网设备：作为访问设备Shell、进行调试和与外部模块（如GPS、GSM模块）通信的主要接口。
    • 工业控制和传统设备：与仍在使用RS-232/RS-485标准的工业设备、路由器、交换机等进行通信。

核心原理与设计

它的核心工作原理是什么？

Serial驱动框架是TTY子系统的一个底层实现。它位于TTY核心和物理UART硬件之间，扮演着承上启下的角色。

  1. 核心数据结构：
    • struct uart_driver: 代表一类UART驱动，例如“8250 UART驱动”。它是一个高层抽象，包含了驱动名称、TTY设备的主/次设备号范围等信息。一个uart_driver可以管理多个物理端口。
    • struct uart_port: 代表一个具体的物理UART端口（例如，COM1口）。这是框架中最核心的数据结构，它包含了操作该物理端口所需的所有信息，如I/O端口地址或内存映射地址、中断号（IRQ）、FIFO大小、时钟频率等。
    • struct uart_ops: 这是一个函数指针表，附加在uart_port中。具体的硬件驱动程序（如8250驱动）必须实现这个表中的函数，例如.startup()（初始化端口）、.shutdown()（关闭端口）、.tx_empty()（检查发送缓冲区是否为空）、.set_termios()（设置波特率、数据位等参数）。
  2. 注册与绑定流程：
    • Serial驱动（如8250_pci驱动）在初始化时，会注册一个uart_driver。
    • 然后，它会探测硬件，为每一个发现的物理UART端口分配并初始化一个uart_port，填充其硬件资源信息和uart_ops函数表，最后调用uart_add_one_port()将其注册到Serial核心。
  3. 与TTY核心的交互：
    • uart_add_one_port()函数会调用tty_port_register_device()，将这个UART端口注册为一个标准的TTY设备（如ttyS0）。
    • 当用户空间程序open("/dev/ttyS0")时，TTY核心会通过一系列调用，最终触发uart_port的.ops->startup()函数，从而对硬件进行上电和初始化。
    • 当用户write()数据时，数据流经TTY核心 -> 线路规程 -> TTY驱动 -> Serial核心，最终调用uart_port的.ops->start_tx()等函数，将数据写入UART的发送FIFO。
    • 当UART硬件收到数据并触发中断时，uart_port的IRQ处理函数会被调用，它会从接收FIFO中读取数据，然后通过tty_insert_flip_string()等函数将数据“喂”给上层的TTY线路规程进行处理。

它的主要优势体现在哪些方面？

  • 高度抽象：将TTY子系统的终端语义与具体的UART硬件操作完全分离。
  • 代码复用：serial_core.c提供了所有UART驱动共享的通用逻辑。
  • 可移植性：设备驱动（如一个需要串口通信的GPS驱动）可以不关心底层使用的是哪种UART芯片，只需与标准的TTY设备交互即可。

它存在哪些已知的劣势、局限性或在特定场景下的不适用性？

  • 绑定于TTY子系统：该框架是为“终端”设备设计的。这意味着所有数据流默认都会经过TTY线路规程的处理（如回显、行编辑）。对于纯粹的、高速的二进制数据传输，这种处理是不必要的开销。虽然可以通过将终端设置为“原始模式”（raw mode）来绕过大部分处理，但这仍然是在TTY框架下的操作。
  • 缺乏协议层：UART本身是一种非常底层的字节流协议，没有内建的寻址、包格式或错误校验机制。这些都需要上层应用自己实现。
  • 速度限制：相比于现代的片上总线（如SPI, I2C），UART/RS-232的速率通常较低。

使用场景

在哪些具体的业务或技术场景下，它是首选解决方案？

  • 内核调试：通过console=ttyS0,115200内核启动参数，将内核的打印信息（printk）重定向到串口，这是进行底层内核开发和调试的“金标准”。
  • 无头（Headless）系统管理：对于没有显示器的服务器或嵌入式设备，串口是获取启动信息和登录Shell的最基本、最可靠的方式。
  • 与外部模块通信：与许多GPS、GSM/LTE、蓝牙模块以及微控制器进行通信，它们通常提供基于AT命令或自定义协议的UART接口。

是否有不推荐使用该技术的场景？为什么？

  • 高速片上通信：不推荐。在同一块PCB上的芯片间（如CPU与一个ADC芯片）进行高速数据传输，应优先使用SPI总线，因为它速度更快，协议更适合这种主从模式。
  • 多设备总线：不推荐。如果需要在一条总线上挂载多个设备（如多个传感器），应使用I2C总线，因为它天生支持设备寻址和多主/多从架构。
  • 网络通信：不推荐。虽然历史上曾有SLIP/PPP协议通过串口进行网络连接，但这已被以太网和Wi-Fi等现代网络技术完全取代。

serial

  • serial 是 Linux 内核中用于管理串口设备的子系统。它负责处理串口通信，包括数据传输、流控制和设备管理等功能。串口设备通常用于与外部设备（如调制解调器、嵌入式系统等）进行通信。
  • 在 Linux 系统中，串口设备通常以 /dev/ttyS* 的形式命名，其中 ttyS0 表示第一个串口设备，ttyS1 表示第二个串口设备，以此类推。对于 USB 串口设备，它们通常以 /dev/ttyUSB* 的形式命名。
  • serial 子系统提供了对串口设备的统一接口，使得用户和应用程序可以方便地进行串口通信。它支持多种串口协议和配置选项，如波特率、数据位、停止位和校验位等。
  • serial 子系统还提供了流控制功能，以确保数据的可靠传输。流控制可以通过硬件（如 RTS/CTS）或软件（如 XON/XOFF）实现。
  • 在 Linux 内核中，serial 子系统与 tty 子系统密切相关，因为串口设备也是终端设备的一种。tty 子系统为串口设备提供了统一的接口，使得用户可以通过标准的终端操作与串口设备进行交互。
  • serial 子系统的实现包括对串口设备驱动程序的支持，这些驱动程序负责与具体的硬件进行交互。常见的串口驱动程序包括 8250 驱动程序（用于传统的 16550 UART）和 usb-serial 驱动程序（用于 USB 串口转换器）。

drivers/tty/serial/serial_ctrl.c

串行总线框架: 控制器核心驱动

此代码片段展示了serial_base虚拟总线框架的控制器侧核心驱动的实现。它的核心作用是为所有”串行控制器”类型的设备提供一个通用的、基础的驱动程序行为。这个驱动本身并不直接与任何具体的硬件交互, 而是作为一个”模板”或”占位符”, 确保了控制器设备在被添加到serial_base总线时, 能够被正确地探测、绑定, 并启用最基本的电源管理功能。

这个驱动的原理可以被理解为一个极简的、框架性的驱动程序实现:

  1. 定义probe和remove函数:

    • serial_ctrl_probe: 这是当一个控制器设备与此驱动成功绑定时被调用的函数。它只做了一件事: pm_runtime_enable(dev)。这会为该设备启用运行时电源管理(Runtime PM)。这意味着即使系统正在运行, 如果内核发现这个控制器设备在一段时间内处于空闲状态, 就会自动将其置于低功耗状态(例如, 关闭其时钟)。
    • serial_ctrl_remove: 这是当设备与驱动解除绑定时调用的函数。它执行probe的反向操作: pm_runtime_disable(dev), 禁用了该设备的运行时电源管理功能。

  2. 定义驱动结构体 (serial_ctrl_driver):

    • 它将上述的probe和remove函数指针打包到一个device_driver结构体中。
    • .name = "ctrl": 这个名称很重要。在serial_base总线的实现中, 很可能会将所有控制器设备的类型名称设置为”ctrl”, 从而确保它们能够与这个通用驱动程序匹配。
    • .suppress_bind_attrs = true: 这个标志禁止在sysfs中为这个驱动创建bind和unbind文件。这通常意味着绑定和解绑是由内核自动管理的, 而不是由用户空间手动触发。

  3. 提供注册/注销接口 (serial_base_ctrl_init, serial_base_ctrl_exit):

    • 这两个函数被serial_base总线框架的主初始化函数serial_base_init调用。
    • serial_base_ctrl_init的作用就是在总线初始化流程中, 将这个通用的serial_ctrl_driver注册到serial_base总线上。
    • serial_base_ctrl_exit则在总线框架退出时负责注销此驱动。

在STM32H750上的应用:
想象一下, 在STM32上有一个驱动程序, 它管理着一个复杂的I3C控制器。这个I3C驱动程序会创建一个platform_device来代表I3C控制器, 并且会将这个设备的bus指针指向serial_base_bus_type, 同时可能将其类型设置为”ctrl”。当这个设备被注册到内核时:

  1. serial_base总线会发现这是一个控制器设备。
  2. 总线核心会将它与已经注册的serial_ctrl_driver进行匹配。
  3. 匹配成功后, serial_ctrl_probe会被调用, 从而为这个I3C控制器设备自动启用了运行时电源管理。
  4. 之后, 具体的I3C驱动程序可以通过其他机制(例如, 子设备或者特定的API)来进一步配置和使用这个控制器, 而这个通用驱动已经为它处理了最基础的设备模型集成和电源管理设置。

这体现了一种驱动分层的设计思想, 将通用的、与设备模型和电源管理相关的模板化代码, 与具体的、跟硬件寄存器交互的逻辑分离开来, 使得代码结构更清晰, 可维护性更高。

drivers/tty/serial/serial_base_bus.c

serial_base总线: 类型匹配与驱动注册

此代码片段定义了serial_base虚拟总线框架的核心骨架和匹配规则。它的根本原理是摒弃Linux设备模型中默认的、基于设备和驱动名称字符串完全匹配的简单机制, 转而实现一种更强大、更灵活的、基于设备类型和驱动名称前缀的自定义匹配逻辑。

这个自定义总线的设计是为了能在一个统一的总线(serial-base)下, 管理两种完全不同角色的设备: 控制器(ctrl)和端口(port), 并确保它们只能与对应角色的驱动程序进行绑定。

核心组件

1. device_type: 设备类型标签

  • 原理: struct device_type 在这里被用作一个高效的”类型标签”。当一个设备被创建时, 它的dev->type指针可以被设置为指向这两个静态结构体之一的地址。因为这两个结构体是静态全局的, 它们的地址在整个内核生命周期中是唯一的。后续的匹配函数可以通过简单的指针比较 (dev->type == &serial_ctrl_type) 来快速、无歧义地判断一个设备的类型, 这比字符串比较要高效得多。

2. serial_base_match: 自定义匹配函数

这是整个自定义总线行为的核心。当有新的设备或驱动注册到serial-base总线上时, 内核总线核心会调用此函数来判断它们是否应该绑定在一起。

  • 原理: 这个函数定义了清晰的、非对称的匹配规则。它不再要求设备和驱动的名字完全一样。
    • 一个被标记为serial_ctrl_type的设备, 只能与名字以”ctrl”为前缀的驱动绑定。例如, 驱动"ctrl-i3c"和"ctrl-rffe"都可以匹配控制器设备, 但驱动"port-sensor"则不能。
    • 同理, 一个被标记为serial_port_type的设备, 只能与名字以”port”为前缀的驱动绑定。
    • 这种机制极大地增强了总线的结构化和灵活性, 允许多个不同类型的具体驱动(如ctrl-i3c, ctrl-rffe)服务于同一类设备(控制器)。

3. bus_type & serial_base_driver_register: 总线定义与驱动注册辅助

这两个部分将上述的组件组装成一个功能性的总线。

serial_base_init: 初始化串行总线基础框架

此函数的核心作用是在Linux内核启动的早期阶段, 注册并初始化一个名为”serial_base”的、自定义的虚拟总线框架。这个框架旨在提供一个标准的抽象层, 用于管理和连接两种不同类型的设备驱动: “串行控制器”(serial base controllers)和”串行端口”(serial base ports)。

该函数的原理可以被理解为一个三步走的、分层式的初始化过程, 这是在Linux设备模型中建立一个新总线系统的标准方法:

  1. 注册总线类型 (bus_register): 这是第一步也是最基础的一步。它向内核的设备模型核心注册一个serial_base_bus_type结构体。这个操作会在sysfs文件系统中创建/sys/bus/serial_base/目录, 并建立一个”事件空间”。之后, 任何声明自己属于这个总线的驱动程序和设备都可以在这个空间内进行匹配和绑定。
  2. 初始化控制器子系统 (serial_base_ctrl_init): 在总线本身存在之后, 它会调用一个辅助函数来初始化该框架的”控制器”部分。控制器可以被理解为这项串行技术能力的提供者。这个初始化过程可能包括为即将注册的控制器驱动程序准备数据结构、链表或创建sysfs子目录(如/sys/bus/serial_base/devices/)。
  3. 初始化端口子系统 (serial_base_port_init): 接着, 它初始化框架的”端口”部分。端口可以被理解为这项串行技术能力的消费者。这个过程与控制器初始化类似, 为端口设备准备所需的环境。

错误处理: 这个函数采用了健壮的、基于goto的错误处理机制。如果在初始化过程的任何一步失败, 它会以相反的顺序精确地撤销所有已经成功完成的步骤, 从而保证系统不会处于一个不一致的、部分初始化的状态。例如, 如果port_init失败, 它会先执行ctrl_exit, 再执行bus_unregister。

在STM32H750上的应用:
虽然这段代码本身是通用的总线框架代码, 但它代表的设计模式在嵌入式系统中非常普遍。在STM32上, “serial_base_controller”可能是一个驱动, 它管理着一个I3C或RFFE控制器硬件外设。而”serial_base_port”则可能是另一个芯片(如一个射频前端模块)的驱动, 这个芯片通过I3C/RFFE总线与STM32相连。serial_base总线就为这两个驱动提供了一个标准的、解耦的交互方式。arch_initcall确保了这个基础框架在任何依赖它的具体设备驱动被探测之前就已经准备就绪, 这对于保证正确的驱动加载和初始化顺序至关重要。
